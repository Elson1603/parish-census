# Parish Census Backend

FastAPI + SQLAlchemy (async) + Alembic, backed by Neon Postgres.

## Stack

- **FastAPI** for the HTTP API
- **SQLAlchemy 2.0** (async, via `asyncpg`) for the ORM
- **Alembic** for migrations
- **uv** for dependency management
- **Neon** (serverless Postgres) as the database

## One-time setup

1. Install dependencies (uv creates `.venv` automatically):

   ```
   uv sync
   ```

2. Create `.env` from the example and fill in your Neon connection string:

   ```
   cp .env.example .env
   ```

   `DATABASE_URL` must be the connection string from your Neon project's **Connect** dialog,
   e.g. `postgresql://user:password@ep-xxxx.aws.neon.tech/dbname?sslmode=require`.

3. Apply migrations:

   ```
   uv run alembic upgrade head
   ```

4. Seed villages + master data lists (occupations, education, church groups, marital status,
   blood groups, special needs) so the admin dropdowns aren't empty:

   ```
   uv run python -m app.seed
   ```

   Safe to re-run — it skips anything that already exists.

## Running the API

```
uv run uvicorn app.main:app --reload --port 8000
```

Interactive docs at `http://localhost:8000/docs`.

The frontend expects `VITE_API_URL=http://localhost:8000` in `frontend/.env` (already set up).

## Project layout

```
app/
  config.py          Settings (reads .env)
  database.py         Async engine/session, translates Neon's sslmode= into an SSL context
  models/census.py     SQLAlchemy models: Village, Family, Member, MasterDataItem
  schemas/census.py    Pydantic request/response schemas (camelCase to match the frontend)
  services/            Shared query builders + village lookup helper
  routers/             One module per resource: villages, families, members, master_data,
                        dashboard, reports, search, intake
  seed.py              Idempotent seed script (see above)
alembic/               Migrations (hand-written, not autogenerated)
```

## Data model notes

- The homepage census intake wizard (`POST /census/intake`) is a lightweight door-to-door
  form: it only collects a member's name, DOB, relation, education, job, church group, and an
  optional remark. It does **not** collect house number, contact number, gender, blood group,
  or sacraments — those columns default to `""` (or `false` for booleans) and can be filled in
  later through the admin "Add Family" / "Add Member" forms, which fully validate those fields.
- Village names typed into the intake wizard are matched case-insensitively against existing
  villages, or created on the fly (`get_or_create_village`), so "Other: ..." free-text entries
  become real villages too.
- Dashboard stats (`GET /dashboard`) are computed live from the `families`/`members` tables —
  nothing is cached or precomputed. `completedFamilies`/`totalExpectedFamilies` are simplified
  to "families with at least one member" / "total families recorded", since there's no separate
  concept of a census target count.
- `POST /drafts/family` and `POST /drafts/member` back the frontend's autosave-while-typing
  hook. They currently just acknowledge the payload without persisting it — the autosave
  feature exists to survive an accidental page refresh, not as a durable draft store. Wire
  these to a real table later if you want drafts to survive across devices/sessions.

## Local smoke testing without Neon

If you want to test against a throwaway local Postgres instead of Neon:

```
docker run -d --name parish-census-db -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=parish_census -p 55432:5432 postgres:16-alpine
```

then set `DATABASE_URL=postgresql://postgres:postgres@localhost:55432/parish_census` in `.env`.
